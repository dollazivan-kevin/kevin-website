Step 01

换首屏动效：从橙色光柱到 DarkVeil

Day 1 搭好了整个网站框架，但首屏的中央大画框效果一直不满意。最开始是蓝色极光+神经网络，后来换成橙色光柱+蚀日球体，看着都还行，但总觉得缺点什么——太刻意，不够"有机"。

今天找到了一个叫 DarkVeil 的开源组件，来自 React Bits 项目。它的底层是一个 CPPN（组合模式生成网络）——本质上是一个用神经网络权重直接生成图像的 WebGL shader。不是预设动画，是数学在实时跑。

什么是 CPPN？
Compositional Pattern Producing Network——一种特殊的神经网络，输入坐标 (x, y)，输出颜色值。把它映射到每个像素就得到一张图。加上时间变量，图就动起来了。完全由矩阵运算驱动，没有任何预制素材。

效果：深邃的有机抽象流动，紫色调为主，随时间缓慢变化。比任何手工绘制的光效都更有"生命感"。

坑 01

OGL 库崩溃，整个网站内容消失

DarkVeil 的原版代码用了一个叫 ogl 的 WebGL 封装库。我在 HTML 里加了 CDN 引入： 

<script src="https://cdn.jsdelivr.net/npm/ogl/dist/ogl.umd.js"></script>

然后 JS 里这样用：

const { Renderer, Program, Mesh, Triangle, Vec2 } = OGL;
问题：OGL 的 UMD 构建加载后，全局变量不叫 OGL，或者根本没有正确暴露。这一行直接抛出 ReferenceError: OGL is not defined。

这个错误本身不可怕，真正可怕的是它在哪里发生的——在一个 <script> 标签里面，而且不在 try-catch 里。

后果：整个 script 块停止执行。包括后面负责给标题加 txt-in 动画类的代码、滚动显示逻辑……全部没跑到。

用户看到的：首屏黑框，标题消失，其他 Section 也不出来。

今天学到的

JS 脚本里一个未被捕获的错误会终止整个 script 块的执行。任何"可能失败"的初始化代码都应该包在 try-catch 里，避免"一个地方坏了，整个页面挂掉"。

坑 02 · 解决方案

不用 OGL，直接写原生 WebGL

与其依赖一个不确定的 CDN 库，不如直接用浏览器内置的 WebGL API——它永远存在，不需要加载任何外部资源。

之前（有问题）

引入 OGL CDN → 全局变量不存在 → ReferenceError → 整个脚本崩溃

修复后

用原生 gl.createShader()、gl.createProgram() 等 API → 无外部依赖 → 稳定

关键代码结构：

// 获取 WebGL 上下文
const gl = cvs.getContext('webgl') || cvs.getContext('experimental-webgl');

// 编译 shader
function mkShader(type, src) {
  const s = gl.createShader(type);
  gl.shaderSource(s, src);
  gl.compileShader(s);
  return s;
}

// 全屏三角形（覆盖整个 clip space，只需 3 个顶点）
gl.bufferData(gl.ARRAY_BUFFER,
  new Float32Array([-1,-1, 3,-1, -1,3]),
  gl.STATIC_DRAW
);

// 动画循环
function loop() {
  gl.uniform1f(uTime, ((performance.now() - t0) / 1000) * speed);
  gl.drawArrays(gl.TRIANGLES, 0, 3);
  requestAnimationFrame(loop);
}

同时在整个 canvas 初始化代码外加了 try-catch，即使 WebGL 出问题，也不会影响页面其他内容的加载。

Step 02

全站英文化 + 改 "AI." 颜色

网站原来是中文。要对外分享，需要切英文。同时把首屏大标题第三行"AI."的颜色从橙色改成 #41BDF4（天蓝色），和整体蓝色基调更统一。

颜色修改

之前是橙色渐变，改成蓝色系：

/* 之前 */
background: linear-gradient(105deg, #FFA040 10%, #FF6010 48%, #FF8830 90%);

/* 现在 */
background: linear-gradient(105deg, #7de3ff 0%, #41BDF4 55%, #1a9fd4 100%);

大标题重写

中文"用审美 / 驯服 / AI."改成英文，保持三行堆叠的视觉节奏：

<!-- 之前 -->
<span class="l1">用审美</span>
<span class="l2">驯服</span>
<span class="l3">AI.</span>

<!-- 现在 -->
<span class="l1">TASTE</span>
<span class="l2">TAMES</span>
<span class="l3">AI.</span>

"TASTE TAMES AI." 三个词，每个都只有一个含义落点，堆在一起读出来是完整的句子。比翻译腔的"用审美驯服 AI"更有力度。

全站一共替换了 30+ 处中文文字，包括导航、正文、日志卡片、关于页、联系区、页脚。用 grep 确认没有遗漏的中文字符。
Step 03

让首页真正显示在根路径 /

网站的设计稿是一个独立的 homepage-v2.html 文件，放在 public/ 目录下。访问 /homepage-v2.html 没问题，但访问根路径 / 还是 Next.js 默认页面。

第一次尝试：next.config.ts rewrites

在配置文件里加：

async rewrites() {
  return [{
    source: '/',
    destination: '/homepage-v2.html',
  }];
}
没用。Next.js 的 App Router page.tsx 优先级高于 rewrites，根路径永远先命中 page.tsx，rewrite 规则根本不触发。

正确解法：直接改 page.tsx

把 src/app/page.tsx 里的内容全部替换：

import { redirect } from 'next/navigation';

export default function Home() {
  redirect('/homepage-v2.html');
}

访问 / 时，Next.js 服务端直接返回 301 跳转到 /homepage-v2.html。

坑 03 · 04 · 05

Cloudflare 部署一直没生效

推了代码，等了好几分钟，刷新还是旧版本。调查下来发现了三个叠在一起的问题：

✕
坑 3：Cloudflare 有一个未合并的 PR 被卡住了

Cloudflare bot 自动开了一个 PR（"Add Cloudflare Workers configuration"），警告说必须先合并这个 PR，否则后续部署可能失败。我们一直没处理，导致新代码也没触发构建。

✕
坑 4：误以为需要去 Vercel 操作

搜索时打开了 Vercel 的页面，完全不是我们的部署平台。网站在 Cloudflare Workers，和 Vercel 没有任何关系。

✓
解决：合并 PR，Cloudflare 开始正常构建

在 GitHub 找到那个 Open 状态的 PR，点 "Confirm merge"。Cloudflare 检测到 merge 后自动触发了新一轮构建，约 4 分钟后部署完成。

今天学到的

Cloudflare 连接 GitHub 仓库后，会自动开一个配置 PR。这个 PR 要及时合并，否则后续所有推送都可能堵塞。以后建新项目时，第一时间合并这个 PR。

Step 04

买域名 + 绑 DNS 全流程

workers.dev 地址在国内不稳定，发给别人经常打不开。需要一个正式域名。

域名选择

目标：kevin.studio。查了一下，2017 年就被人注册了。最终选了 kevindesign.vip，在阿里云用支付宝购买，约 ¥XX/年。

完整连接流程（图文版）

  • 1
    在阿里云买好域名 kevindesign.vip
  • 2
    打开 Cloudflare → Domains → Onboard a domain，输入 kevindesign.vip
  • 3
    Cloudflare 给出两个 Nameserver 地址：arch.ns.cloudflare.com 和 isla.ns.cloudflare.com
  • 4
    回到阿里云 → 域名管理 → DNS修改，把原来的 dns11.hichina.com / dns12.hichina.com 替换成 Cloudflare 的两个地址
  • 5
    回 Cloudflare → Workers & Pages → kevin-website → Settings → Domains & Routes → Add → Custom Domain → 填入 kevindesign.vip
  • 6
    等待 DNS 生效（通常 10-30 分钟，最长 48 小时）
结果：https://kevindesign.vip 现在可以正常访问，直接显示首页设计稿，任何人都能打开。
总结

今天学到什么

今天的进展比 Day 1 复杂得多，主要因为开始跟基础设施打交道——部署、DNS、域名。几个关键认知：

技术层面

  • ①
    不要用不确定的 CDN 依赖，尤其是非主流库。全局变量名不对、加载失败、版本问题，任何一个都能让页面挂掉。能用原生 API 就用原生。
  • ②
    JS 错误会"传染"整个 script 块。重要的初始化代码要包 try-catch，让一个功能的失败不要拖垮整个页面。
  • ③
    Next.js 的 page.tsx 优先级高于 rewrites。要覆盖根路径，直接改 page.tsx，不要绕弯子。

流程层面

  • ①
    Cloudflare 接入 GitHub 后会自动开一个配置 PR，第一时间合并它，否则后续推送不会触发构建。
  • ②
    DNS 修改后要等待生效，这段时间无法加速。阿里云写的是 24-48 小时，实际通常 30 分钟内。
  • ③
    workers.dev 地址不稳定，只适合开发调试，对外分享必须用自己的域名。
